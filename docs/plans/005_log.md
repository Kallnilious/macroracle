# Phase 5 — Daily Food Log & Remaining Macros

Status: **APPROVED**

## Goal

Users log food entries (food ID + grams consumed + timestamp). The daily summary endpoint
returns total macros consumed today vs. their profile targets and — critically — remaining
macros. This is the "what have I eaten" data layer that feeds the decision engine in Phase 6.

## Scope

### IN

- `log_entries` table with full CRUD (create, read, delete — no update; delete and re-log instead).
- `POST /log` — create a log entry.
- `GET /log/today` — all of today's entries with per-entry computed macros.
- `GET /log/summary` — daily totals vs. profile targets + remaining macros.
- `DELETE /log/:id` — remove a log entry.
- `GET /log?date=YYYY-MM-DD` — retrieve entries for a specific past date.
- Frontend: `LogEntryForm` (food search + grams input) and `TodaySummary` (macro progress bars).
- Unit tests in `packages/core` for macro calculation.
- Integration tests against `macroracle_test` DB for all endpoints.
- Soft-delete on `foods` table (add `deleted_at` column) so log entries survive food removal.

### OUT

- Editing a log entry (delete and re-log).
- Bulk import / CSV upload.
- Meal-level grouping (breakfast / lunch / dinner) — can add later as a tag.
- Calorie-only logging without a `food_id` (requires a freeform "quick log" — future phase).
- Push notifications or reminders to log.
- Timezone UI (the frontend does not yet have a TZ picker; default UTC for now).

## Data model

### Migration: `005_log.sql`

```sql
-- Soft-delete support on foods
ALTER TABLE foods ADD COLUMN IF NOT EXISTS deleted_at TIMESTAMPTZ DEFAULT NULL;

-- Log entries
CREATE TABLE log_entries (
  id          UUID PRIMARY KEY DEFAULT gen_random_uuid(),
  user_id     UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  food_id     UUID NOT NULL REFERENCES foods(id),  -- no ON DELETE CASCADE; see edge cases
  grams       NUMERIC(8,2) NOT NULL CHECK (grams > 0),
  logged_at   TIMESTAMPTZ NOT NULL DEFAULT NOW(),
  notes       TEXT,
  created_at  TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

CREATE INDEX log_entries_user_date ON log_entries (user_id, logged_at);
```

**Rationale for `REFERENCES foods(id)` without CASCADE:** We soft-delete foods so existing
log entries are never orphaned. Deleting a log entry removes the pointer; the food row stays
with `deleted_at` set.

### Derived columns (computed on read, never stored)

| Name | Formula |
|---|---|
| `calories` | `food.calories_per_100g * grams / 100` |
| `protein_g` | `food.protein_per_100g * grams / 100` |
| `carbs_g` | `food.carbs_per_100g * grams / 100` |
| `fat_g` | `food.fat_per_100g * grams / 100` |

## API surface

All endpoints require `Authorization: Bearer <jwt>`. All responses are JSON.

---

### `POST /log`

Create a new log entry for the authenticated user.

**Request body:**
```json
{
  "food_id": "uuid",
  "grams": 150,
  "logged_at": "2026-06-21T13:00:00Z",  // optional; defaults to NOW()
  "notes": "post-workout"               // optional
}
```

**Response `201`:**
```json
{
  "id": "uuid",
  "food_id": "uuid",
  "food_name": "Chicken Breast",
  "grams": 150,
  "logged_at": "2026-06-21T13:00:00Z",
  "macros": {
    "calories": 247.5,
    "protein_g": 46.5,
    "carbs_g": 0,
    "fat_g": 5.4
  }
}
```

**Errors:**
- `401` — missing/invalid JWT.
- `404 food_not_found` — food_id does not exist or is soft-deleted.
- `422 no_profile` — user has no profile (cannot compute remaining macros for summary, but entry
  is still accepted — return 422 with `"message": "Set your profile first to see remaining macros"`,
  log entry is NOT saved; require profile before logging.

**Decision:** Require profile before any logging. This keeps the summary endpoint always
computable. Revisit if users want to log before setting a profile.

---

### `GET /log/today`

Returns all log entries for the authenticated user for today (UTC date, or user-provided
`tz` query param — see Edge cases).

**Query params:** `tz` (optional IANA timezone string, e.g. `America/New_York`).

**Response `200`:**
```json
{
  "date": "2026-06-21",
  "entries": [
    {
      "id": "uuid",
      "food_id": "uuid",
      "food_name": "Chicken Breast",
      "food_deleted": false,
      "grams": 150,
      "logged_at": "2026-06-21T13:00:00Z",
      "notes": null,
      "macros": {
        "calories": 247.5,
        "protein_g": 46.5,
        "carbs_g": 0,
        "fat_g": 5.4
      }
    }
  ]
}
```

If a food has been soft-deleted, `food_deleted: true` and the food name is preserved from
the row (names are stored on `foods` and the row is not removed, only `deleted_at` is set).

---

### `GET /log/summary`

Returns macro totals vs. targets and remaining macros for today (or a given date via `?date=`).

**Query params:** `date` (optional `YYYY-MM-DD`), `tz` (optional IANA TZ).

**Response `200`:**
```json
{
  "date": "2026-06-21",
  "targets": {
    "calories": 2200,
    "protein_g": 165,
    "carbs_g": 220,
    "fat_g": 73
  },
  "consumed": {
    "calories": 247.5,
    "protein_g": 46.5,
    "carbs_g": 0,
    "fat_g": 5.4
  },
  "remaining": {
    "calories": 1952.5,
    "protein_g": 118.5,
    "carbs_g": 220,
    "fat_g": 67.6
  },
  "pct_complete": {
    "calories": 11.25,
    "protein_g": 28.18,
    "carbs_g": 0,
    "fat_g": 7.40
  }
}
```

**Errors:**
- `422 no_profile` — user has no profile; cannot compute targets.

---

### `DELETE /log/:id`

Remove a specific log entry. User can only delete their own entries.

**Response `204`:** No body.

**Errors:**
- `404` — entry not found or belongs to another user.

---

### `GET /log?date=YYYY-MM-DD`

Returns entries + per-entry macros for a specific past (or future) date, same shape as
`GET /log/today` but for the given date.

**Response:** Same structure as `GET /log/today` with `date` reflecting the requested date.

## Algorithms

All macro calculation lives in `packages/core/src/macros.ts` so the RN app reuses it unchanged.

### Per-entry macro calculation

```
function computeEntryMacros(food: FoodNutrients, grams: number): EntryMacros {
  const factor = grams / 100
  return {
    calories:  round2(food.calories_per_100g  * factor),
    protein_g: round2(food.protein_per_100g   * factor),
    carbs_g:   round2(food.carbs_per_100g     * factor),
    fat_g:     round2(food.fat_per_100g       * factor),
  }
}
// round2: Math.round(value * 100) / 100
```

### Daily totals

```
function sumMacros(entries: EntryMacros[]): EntryMacros {
  return entries.reduce((acc, e) => ({
    calories:  round2(acc.calories  + e.calories),
    protein_g: round2(acc.protein_g + e.protein_g),
    carbs_g:   round2(acc.carbs_g   + e.carbs_g),
    fat_g:     round2(acc.fat_g     + e.fat_g),
  }), { calories: 0, protein_g: 0, carbs_g: 0, fat_g: 0 })
}
```

### Remaining macros

```
function computeRemaining(targets: MacroTargets, consumed: EntryMacros): EntryMacros {
  return {
    calories:  round2(targets.calories  - consumed.calories),
    protein_g: round2(targets.protein_g - consumed.protein_g),
    carbs_g:   round2(targets.carbs_g   - consumed.carbs_g),
    fat_g:     round2(targets.fat_g     - consumed.fat_g),
  }
  // Remaining values CAN be negative (user went over). Engine must handle negative remaining.
}
```

### Percentage complete

```
pct = (consumed[macro] / targets[macro]) * 100
// Cap display at 100% (but store raw value so engine knows macro is over-target)
```

### SQL query for today's entries

```sql
SELECT
  le.id,
  le.food_id,
  f.name        AS food_name,
  f.deleted_at IS NOT NULL AS food_deleted,
  le.grams,
  le.logged_at,
  le.notes,
  f.calories_per_100g,
  f.protein_per_100g,
  f.carbs_per_100g,
  f.fat_per_100g
FROM log_entries le
JOIN foods f ON f.id = le.food_id
WHERE le.user_id = $1
  AND DATE(le.logged_at AT TIME ZONE $2) = $3  -- $2 = tz string, $3 = YYYY-MM-DD
ORDER BY le.logged_at ASC;
```

Macro multiplication is done in the application layer (TypeScript), not in SQL, so the
formula lives in `packages/core` and is reusable.

## Edge cases & failure modes

| Scenario | Handling |
|---|---|
| User has no profile | `POST /log` returns `422 no_profile`; prompt user to complete profile |
| Food is soft-deleted after logging | Entry remains; `food_deleted: true` in response; macros still computed from food row |
| `grams` = 0 or negative | DB `CHECK (grams > 0)` + API validation before insert; return `422 invalid_grams` |
| `grams` extremely large (e.g. 99999) | No hard cap in schema; frontend caps input at 9999g with a warning |
| `food_id` does not exist | Foreign key violation caught; return `404 food_not_found` |
| Negative remaining macros | Allowed — user exceeded target; UI shows red progress bar; engine will handle in Phase 6 |
| Timezone: "today" is ambiguous | Accept `tz` query param (IANA string); default to UTC if absent; document clearly |
| Clock skew: `logged_at` in the future | Accept it — user may be pre-logging; no validation on future timestamps |
| Duplicate rapid submits | Idempotency not required; two identical requests create two entries (correct behaviour) |
| Very large calorie database food (e.g. oil, 900 kcal/100g) with 500g logged | Math is correct; no cap. UI should make grams feel large so user notices. |
| Concurrent deletes (two tabs) | Last DELETE wins; if entry already gone, return `404` gracefully |

## Test plan

### Unit tests — `packages/core/src/macros.test.ts`

1. `computeEntryMacros` with 100g of a known food → returns exactly the per-100g values.
2. `computeEntryMacros` with 150g → all values scaled by 1.5, rounded to 2dp.
3. `computeEntryMacros` with 33.3g → correct rounding (no floating-point drift).
4. `sumMacros` with empty array → zeros.
5. `sumMacros` with 3 entries → correct sum.
6. `computeRemaining` with consumed < targets → positive remaining.
7. `computeRemaining` with consumed > targets → negative remaining (not clamped).

### Integration tests — `apps/api/src/__tests__/log.test.ts`

Setup: create user, set profile, insert a food (USDA or personal).

1. `POST /log` with valid food_id + grams → 201, response includes computed macros.
2. `GET /log/today` after logging → entry appears with correct macros.
3. `GET /log/summary` → consumed matches logged entry; remaining = targets − consumed.
4. Log two entries, `GET /log/summary` → consumed is sum of both; remaining updated.
5. `DELETE /log/:id` → 204; `GET /log/today` no longer includes that entry.
6. `DELETE /log/:id` with another user's entry → 404.
7. `POST /log` with nonexistent food_id → 404.
8. `POST /log` without profile → 422 `no_profile`.
9. `GET /log?date=2026-01-01` → returns entries for that date, not today's.
10. Log entry with `logged_at` = yesterday; `GET /log/today` → NOT included; `GET /log?date=<yesterday>` → included.

Run: `npm test` from repo root (Vitest workspace config picks up all test files).

## Migration / rollback

**Apply:**
```
node apps/api/dist/db/migrate.js up
# runs 005_log.sql: ALTER TABLE foods ADD deleted_at; CREATE TABLE log_entries
```

**Rollback:**
```sql
-- 005_log_rollback.sql
DROP TABLE IF EXISTS log_entries;
ALTER TABLE foods DROP COLUMN IF EXISTS deleted_at;
```

The rollback script lives at `db/migrations/005_log_rollback.sql`. Run manually; the
migration runner does not auto-rollback.

**Data risk:** Dropping `log_entries` destroys all log data. In production, prefer disabling
the feature over dropping the table.

## Open questions

1. **Soft-delete vs. cascade on foods:** We chose soft-delete so log history is preserved.
   Is a "deleted" food's macros still trustworthy (user may have edited it before deleting)?
   Probably yes — snapshot the macros at log time in a future phase if auditing matters.

2. **Timezone handling:** Defaulting to UTC is simple but means a user logging at 11pm EST
   sees their entry on the "wrong" day. Should we prompt for timezone in the profile (Phase 2
   revisit) rather than per-request query param? Deferred to Phase 9 UI work.

3. **Editing log entries:** "Delete and re-log" is deliberately simple for v1. Is it
   frustrating enough to need an edit endpoint in Phase 5, or can it wait?

4. **Grams cap:** No hard cap in the DB. Should the API reject unreasonably large grams
   (say > 5000g)? Probably a frontend concern for now.

5. **`logged_at` default:** The client sends the current time; the server accepts it. If the
   client clock is wrong, entries land on the wrong day. Should the server use `NOW()` always?
   Trade-off: flexibility vs. correctness. Current decision: trust client timestamp.
