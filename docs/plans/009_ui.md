# Phase 9 — Frontend Polish & React Native Prep

Status: **APPROVED**

## Goal

Complete the React frontend by wiring up all features with proper routing, shared
components, consistent error handling, and loading states. Simultaneously validate and
enforce the architectural boundary that will make a React Native port clean: ensure
`packages/core` is DOM-free and Node.js-free, all business logic lives in core (not in
components), API calls are isolated in a typed API layer, and platform-specific code
is quarantined behind an interface.

This phase ships a production-quality web app. It does not write a single line of React
Native code — that is a separate project. What it does is eliminate the friction that
would make a future port painful.

## Scope

### IN

- All routing: react-router-dom v6, five main routes + auth routes.
- `AuthContext` and `ProfileContext` — React Contexts replacing ad-hoc prop drilling.
- `ProtectedRoute` wrapper for authenticated routes.
- `NavBar`, `LoadingSpinner`, `MacroProgressBar`, `ErrorBoundary` shared components.
- CSS Modules for all styling (no Tailwind — keep dependencies lean).
- All seven pages built and wired: `OraclePage`, `LogPage`, `FoodsPage`, `ProfilePage`,
  `WeightPage`, `LoginPage`, `RegisterPage`.
- Typed API layer: one file per resource in `apps/web/src/api/`.
- Platform storage abstraction: `apps/web/src/platform/storage.ts`.
- RN-readiness audit of `packages/core` (automated check in CI).
- Vitest + `@testing-library/react` tests for all page components.
- Accessibility basics: semantic HTML, keyboard navigation, ARIA labels on icon buttons.
- Environment-variable-driven API base URL (`VITE_API_URL`).

### OUT

- React Native code (deferred entirely — this phase only *prepares* for it).
- PWA / service workers / offline mode.
- Tailwind CSS (decision: CSS Modules only in Phase 9; Tailwind can be adopted later if
  team alignment exists).
- Dark mode.
- Internationalisation (i18n).
- Animation / transitions beyond simple CSS.
- End-to-end tests (Playwright / Cypress) — the integration + component tests are
  sufficient for Phase 9.
- Advanced accessibility (WCAG AA full audit) — basics only.
- Performance optimisation (code splitting, lazy loading) — add in Phase 10 if needed.

## Data model

No new tables. No new API endpoints. This phase is frontend-only.

## API surface

No new endpoints. No schema changes.

The API layer (`apps/web/src/api/`) wraps existing endpoints in typed functions so
components never use raw `fetch` directly. This is the only new "surface" in Phase 9.

```
apps/web/src/api/
  auth.ts      — login(), register(), logout()
  foods.ts     — searchFoods(), getFood(), createFood(), updateFood(), deleteFood()
  log.ts       — createLogEntry(), getTodayLog(), getLogSummary(), deleteLogEntry(), getLogByDate()
  oracle.ts    — askOracle(), getLastOracle()
  profile.ts   — getProfile(), upsertProfile()
  weight.ts    — logWeight(), getWeightHistory(), triggerCalibration(), getCalibrationHistory()
```

Each function signature follows the pattern:
```typescript
async function login(email: string, password: string): Promise<AuthResponse>
// throws ApiError (custom class) on non-2xx; ApiError has { status, code, message }
```

No `axios`. Use the browser `fetch` API with a thin shared `apiFetch` wrapper that:
1. Reads the base URL from `import.meta.env.VITE_API_URL`.
2. Attaches `Authorization: Bearer <token>` from `platform/storage.ts`.
3. Parses JSON response.
4. Throws `ApiError` on non-2xx.

## Frontend

### Routing

```typescript
// apps/web/src/App.tsx
import { BrowserRouter, Routes, Route } from 'react-router-dom'

<BrowserRouter>
  <NavBar />
  <Routes>
    <Route path="/login"    element={<LoginPage />} />
    <Route path="/register" element={<RegisterPage />} />
    <Route element={<ProtectedRoute />}>
      <Route path="/"        element={<OraclePage />} />
      <Route path="/log"     element={<LogPage />} />
      <Route path="/foods"   element={<FoodsPage />} />
      <Route path="/profile" element={<ProfilePage />} />
      <Route path="/weight"  element={<WeightPage />} />
    </Route>
    <Route path="*" element={<NotFoundPage />} />
  </Routes>
</BrowserRouter>
```

`ProtectedRoute` reads `AuthContext.user`; redirects to `/login` if unauthenticated,
preserving the intended destination via `state: { from: location }`.

---

### State architecture

**Two Contexts only. No Redux.**

```typescript
// AuthContext: { user: User | null, login, logout, loading }
// ProfileContext: { profile: Profile | null, refetch }
```

- `AuthContext` is the outermost provider. Reads token from `platform/storage.ts` on mount
  to restore session. Sets `loading: true` until the token is validated or rejected.
- `ProfileContext` is nested inside auth. Fetches profile on auth change.
- All other data (log entries, foods, weight) is **local component state** with `useEffect`
  fetches. Data is simple enough — global state would be premature optimisation.

---

### Shared components

**`NavBar`** (`apps/web/src/components/NavBar.tsx`)
- Horizontal nav with links: Oracle (home), Log, Foods, Profile, Weight.
- Shows user email in corner; Logout button.
- Hidden on `/login` and `/register`.
- Active link highlighted via `NavLink` from react-router-dom.

**`ProtectedRoute`** (`apps/web/src/components/ProtectedRoute.tsx`)
- Renders `<Outlet />` if authenticated.
- Shows `<LoadingSpinner />` while `AuthContext.loading` is true.
- Redirects to `/login` if not authenticated.

**`LoadingSpinner`** (`apps/web/src/components/LoadingSpinner.tsx`)
- Simple CSS-animated spinner. Accepts `size?: 'sm' | 'md' | 'lg'` and `label?: string`
  (for `aria-label`).

**`MacroProgressBar`** (`apps/web/src/components/MacroProgressBar.tsx`)
- Props: `label`, `consumed`, `target`, `unit` (g / kcal).
- Renders a labelled progress bar.
- Color: green (0–80%), amber (80–100%), red (> 100%).
- Shows: `consumed / target unit` and `remaining unit remaining`.
- No third-party chart library — pure CSS + `<div>` width trick.

**`ErrorBoundary`** (`apps/web/src/components/ErrorBoundary.tsx`)
- Class component (required by React's error boundary API).
- Catches render errors; shows a friendly message + "Reload" button.
- Wrap each page with its own `ErrorBoundary` so one broken page doesn't crash the nav.

### Pages

**`LoginPage`** — email + password form; calls `auth.login()`; on success stores token,
sets `AuthContext.user`, navigates to `state.from ?? '/'`.

**`RegisterPage`** — email + password + confirm form; calls `auth.register()`;
on success auto-logs in.

**`OraclePage`** (Phase 6 component, now fully wired)
- Fetches `getLogSummary()` on mount to show remaining macros strip.
- `OracleInput` + `ProphecyCard` list.
- "Log this" on a ProphecyCard navigates to `/log` with pre-filled state
  (`{ food_id, grams }` via `useNavigate` + `state`).
- "What did I say last time?" accordion that calls `getLastOracle()`.

**`LogPage`** (Phase 5 component, now fully wired)
- `TodaySummary` at top (macro progress bars).
- `LogEntryForm` below.
- List of today's entries with delete buttons.
- Date picker to view past days (calls `getLogByDate(date)`).
- Pre-fill support: reads `useLocation().state` to pre-fill from oracle "Log this".

**`FoodsPage`** (Phase 3 component, now fully wired)
- Search bar: calls `searchFoods(q)`, shows merged PERSONAL + USDA + OFF results.
- "Add Food" button → modal or inline form for creating a personal food.
- Each personal food has "Edit" and "Delete" buttons (USDA and OFF are read-only).
- Tags editing for personal foods (Phase 6 `PATCH /foods/:id/tags`).

**`ProfilePage`** (Phase 2 component, now fully wired)
- Displays current profile: height, weight, age, sex, activity, goal.
- Macro targets (read-only, computed from TDEE).
- Edit mode: form pre-filled from profile; submits to `upsertProfile()`.
- Shows `tdee_override` if set ("Currently using calibrated TDEE: 2464 kcal/day"
  + "Reset to calculated" button that clears the override).

**`WeightPage`** (Phase 7 component, now fully wired)
- `WeightLogForm` + weight entry list.
- `WeightChart` sparkline.
- `CalibrationSummary` + "Calibrate Now" button.

**`NotFoundPage`** — simple 404 with a link home.

### Platform storage abstraction

```typescript
// apps/web/src/platform/storage.ts
// In React Native, this file would be replaced with AsyncStorage-backed equivalents.
export const storage = {
  getToken: (): string | null => localStorage.getItem('auth_token'),
  setToken: (t: string): void  => localStorage.setItem('auth_token', t),
  clearToken: (): void         => localStorage.removeItem('auth_token'),
}
```

`localStorage` is only accessed through this module. The `apiFetch` wrapper imports
`storage.getToken()`. When porting to RN, only this file needs to change.

## Algorithms

No new algorithms. Phase 9 moves existing logic from components into its canonical location
(core or API layer) and validates the boundaries hold.

**Token refresh:** Not implemented. JWT expiry should be generous (e.g., 30 days) for v1.
A 401 response from any API call redirects to `/login` and clears the token. True refresh
tokens are a Phase 10 concern.

**Form validation:** Client-side validation is done with plain JS (no library). Each form
field validates on blur; shows inline error message. Required, type, range checks only.
Server-side validation is the source of truth; client-side is UX-only.

## Edge cases & failure modes

| Scenario | Handling |
|---|---|
| JWT expired mid-session | Any `ApiError` with `status: 401` → `logout()` + redirect `/login` |
| Network offline | `fetch` rejects; `ApiError` with `status: 0`; show "No connection — check your network" toast |
| Profile not set, user visits `/` | `OraclePage` renders a "Set your profile first" banner with link to `/profile` |
| `ProtectedRoute` flash of unauthenticated content | `AuthContext.loading` shows `<LoadingSpinner>` until token validated; no flash |
| User navigates away mid-form | No unsaved-changes warning in Phase 9 — acceptable for v1 |
| Register with already-taken email | `ApiError { code: 'email_taken' }` → inline error on email field |
| Browser without `localStorage` (private mode in some browsers) | `storage.getToken()` returns null → treated as unauthenticated; user logs in each session |
| `VITE_API_URL` not set | `apiFetch` falls back to `/api` (relative URL, works when web and API are co-hosted) |
| Component render error | `ErrorBoundary` catches, shows fallback; other pages still work |
| Long food name overflows `ProphecyCard` | CSS `overflow: hidden; text-overflow: ellipsis` on name element |
| Very small screen (< 320px) | Minimum supported width is 375px (iPhone SE). Not tested below that. |

## Migration / rollback

Frontend-only phase. No DB changes.

**Rollback:** `git revert` the Phase 9 commits. The backend is unchanged; any prior
frontend version continues to work against the API. Risk: zero data loss.

**Deployment:** Phase 9 ships the static build to wherever the web app is hosted. If
using the sandbox, `npm run build` in `apps/web/` produces `dist/` which is served by
the Express API as static files (or a separate static server — configure in Phase 9).

Add to `apps/api/src/index.ts` for co-hosting (optional but convenient):
```typescript
import path from 'path'
import { fileURLToPath } from 'url'
const __dirname = path.dirname(fileURLToPath(import.meta.url))
app.use(express.static(path.join(__dirname, '../../web/dist')))
app.get('*', (_req, res) => res.sendFile(path.join(__dirname, '../../web/dist/index.html')))
```

## Open questions

1. **CSS Modules vs. Tailwind:** This plan chooses CSS Modules for Phase 9 to keep
   dependencies minimal. If the team wants Tailwind for productivity, the decision should
   be made before Phase 9 starts — adding Tailwind mid-phase is disruptive. Raise before
   kickoff.

2. **Token refresh:** Current plan is long-lived JWT (30 days), no refresh tokens.
   If the security posture needs to tighten (shorter JWTs + refresh), implement in Phase 10.
   Document the trade-off: longer tokens are less secure but far simpler to implement.

3. **`VITE_API_URL` in production:** Where does the API live? Same origin (proxied by
   the web server), or a separate domain? The relative URL fallback (`/api`) works for
   same-origin. If different origins, CORS must be enabled on the API (currently it is
   not, or only allows localhost). Confirm deployment architecture before Phase 9 ships.

4. **Accessibility audit depth:** Phase 9 targets keyboard navigation and ARIA labels on
   icon buttons. Full WCAG AA is deferred. Is this acceptable for the target user base?
   Confirm before shipping publicly.

5. **`ErrorBoundary` granularity:** Wrapping each page in its own boundary is fine.
   Should there be a top-level boundary too (in case NavBar itself crashes)? Probably yes
   — add a top-level boundary around the whole app that shows a "something went very wrong"
   full-page fallback.

6. **Shared types between web and API:** Currently types may be duplicated. Phase 9
   enforces that `packages/core/src/types.ts` is the single source of truth. The API
   imports from core; so does the web. This requires `packages/core` to be a proper
   TypeScript project reference in both `tsconfig.json` files. Verify the monorepo
   build setup supports this before implementing.
