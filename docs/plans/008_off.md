# Phase 8 — Community Layer (Open Food Facts)

Status: **APPROVED**

## Goal

Add Open Food Facts (OFF) as the community layer in the food resolver. Food search now
queries in priority order: PERSONAL → USDA → OFF (first non-empty result set wins per
food item; deduplication merges overlapping results). OFF data is read-only — no writes,
no contributions, no editing. This expands the food catalogue dramatically without any
additional cost (OFF is free and public).

## Scope

### IN

- `off_foods_cache` table with a 7-day TTL.
- OFF API integration (search + barcode lookup) behind the backend — never called from the client.
- Extended `GET /foods/search` that merges PERSONAL + USDA + OFF results.
- `GET /off/food/:barcode` — barcode lookup stub (for a future scan feature).
- OFF nutrient mapper: extract the four macros from OFF's `nutriments` object.
- Cache-read-first strategy: check `off_foods_cache` before hitting the OFF API.
- Graceful degradation: if OFF is unreachable, return PERSONAL + USDA results only, no error.
- Unit tests for the OFF nutrient mapper and resolver priority logic.
- Integration tests for cache hit/miss and graceful degradation.

### OUT

- Editing or contributing data back to OFF.
- User-facing "report bad data" for OFF entries.
- Barcode scanning UI (Phase 8 stubs the endpoint; the camera/scan UI is a future phase).
- Fuzzy deduplication beyond lowercase-trim name matching (ML-based dedup is out of scope).
- Importing OFF data in bulk into our DB.
- Caching USDA results (USDA has a higher rate limit and stricter TOS around caching —
  revisit separately if needed).
- Authentication with the OFF API (OFF allows anonymous reads at moderate volume).

## Data model

### Migration: `008_off.sql`

```sql
CREATE TABLE off_foods_cache (
  barcode         TEXT PRIMARY KEY,         -- EAN-13 or UPC-A barcode string
  off_product_id  TEXT NOT NULL,            -- OFF internal _id field
  name            TEXT NOT NULL,
  brand           TEXT,
  nutrients       JSONB NOT NULL,           -- raw OFF nutriments object, for forward-compat
  calories_per_100g  NUMERIC(8,2),
  protein_per_100g   NUMERIC(8,2),
  carbs_per_100g     NUMERIC(8,2),
  fat_per_100g       NUMERIC(8,2),
  cached_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Search cache: keyed by normalised query string
-- Separate from barcode cache; holds arrays of matching barcodes
CREATE TABLE off_search_cache (
  query_key       TEXT PRIMARY KEY,         -- normalised(lowercase, trimmed) search term
  barcodes        TEXT[] NOT NULL,          -- ordered list of barcodes for this query
  cached_at       TIMESTAMPTZ NOT NULL DEFAULT NOW()
);

-- Cleanup function: entries older than 7 days are stale
-- Run this as part of the search handler (lazy eviction) or via a scheduled job
-- No separate cron in Phase 8; lazy eviction on cache read is sufficient.
```

**Why separate `off_foods_cache` and `off_search_cache`?** A barcode lookup caches one
product; a search caches a list of barcodes. Products can be reused across searches.
This avoids duplicating full product records per search term.

**TTL enforcement:** On cache read, check `cached_at + INTERVAL '7 days' > NOW()`. If stale,
delete the row and fetch fresh. Lazy eviction — no background cleanup process needed in Phase 8.

### No changes to `foods` table

OFF results are served from the cache, not inserted into `foods`. This preserves the
PERSONAL / USDA / OFF distinction and avoids polluting the personal food catalogue.

## API surface

### Extended: `GET /foods/search?q=<term>`

**No change to request shape.** Response now includes an `off` result set in addition to
`personal` and `usda` results, and a merged top-level `results` array.

**Response `200`:**

**Ordering:** PERSONAL results first, then USDA, then OFF. Within each source, results
are ordered by relevance (name match score). Deduplication removes OFF results whose
normalised name is identical to a PERSONAL or USDA result already in the list.

**Degraded response** (OFF unreachable):
```json
{
  "results": [ /* personal + usda only */ ],
  "sources_queried": ["personal", "usda"],
  "off_error": "Open Food Facts unavailable — showing personal and USDA results only."
}
```

### New: `GET /off/food/:barcode`

Look up a single product by barcode. Checks cache first; falls back to OFF API.

**Response `200`:**
```json
{
  "barcode": "737628064502",
  "name": "Organic Chicken Breast",
  "brand": "Whole Foods",
  "calories_per_100g": 110,
  "protein_per_100g": 23.0,
  "carbs_per_100g": 0,
  "fat_per_100g": 2.0,
  "from_cache": false
}
```

**Response `404`:** Barcode not found in OFF or cache.

**Response `422 incomplete_nutrients`:** OFF product found but one or more macro fields
are missing. Return 422 with available fields and a `missing_fields` array, so the client
can prompt the user to fill in the gaps before logging.

---

### Internal (not user-facing): `DELETE /off/cache` (admin only, for testing)

Clears the OFF search and product caches. Requires an `X-Admin-Key` header matching
`process.env.ADMIN_KEY`. Used in integration tests only.

## Algorithms

### Resolver priority — updated layered search

```typescript
// apps/api/src/foods/resolver.ts  (extended from Phase 4)
async function searchFoods(query: string, userId: string): Promise<SearchResult[]> {
  const q = query.toLowerCase().trim()

  // Layer 1: PERSONAL (always queried)
  const personal = await searchPersonalFoods(q, userId)

  // Layer 2: USDA (always queried; Phase 4 integration)
  const usda = await searchUsdaFoods(q)

  // Layer 3: OFF (try cache first; fall back to API; catch errors silently)
  let off: OffResult[] = []
  let offError: string | null = null
  try {
    off = await searchOffFoods(q)
  } catch (err) {
    offError = 'Open Food Facts unavailable'
    // Do NOT rethrow — degraded mode is acceptable
  }

  // Deduplication: remove OFF results whose normalised name matches a personal or USDA result
  const existingNames = new Set([
    ...personal.map(f => normalise(f.name)),
    ...usda.map(f => normalise(f.name)),
  ])
  const deduped = off.filter(f => !existingNames.has(normalise(f.name)))

  return [
    ...personal.map(f => ({ ...f, source: 'personal' })),
    ...usda.map(f => ({ ...f, source: 'usda' })),
    ...deduped.map(f => ({ ...f, source: 'off' })),
  ]
}

function normalise(name: string): string {
  return name.toLowerCase().trim().replace(/\s+/g, ' ')
}
```

### OFF search — cache-first

```typescript
async function searchOffFoods(q: string): Promise<OffResult[]> {
  // 1. Check search cache
  const cached = await db.query(
    `SELECT barcodes, cached_at FROM off_search_cache
     WHERE query_key = $1 AND cached_at > NOW() - INTERVAL '7 days'`,
    [q]
  )
  if (cached.rows.length > 0) {
    const barcodes: string[] = cached.rows[0].barcodes
    return resolveBarcodesFromCache(barcodes)
  }

  // 2. Cache miss: hit the OFF API
  const url = `https://world.openfoodfacts.org/cgi/search.pl`
    + `?search_terms=${encodeURIComponent(q)}&json=1&page_size=10&fields=code,product_name,brands,nutriments`
  const res = await fetch(url, { signal: AbortSignal.timeout(5000) })  // 5s timeout
  if (!res.ok) throw new Error(`OFF search failed: ${res.status}`)

  const data = await res.json()
  const products: OffProduct[] = (data.products ?? []).filter(hasCompleteNutrients)

  // 3. Cache individual products
  for (const p of products) {
    await upsertOffProduct(p)
  }

  // 4. Cache the search result (list of barcodes)
  const barcodes = products.map(p => p.code)
  await db.query(
    `INSERT INTO off_search_cache (query_key, barcodes, cached_at)
     VALUES ($1, $2, NOW())
     ON CONFLICT (query_key) DO UPDATE SET barcodes = EXCLUDED.barcodes, cached_at = NOW()`,
    [q, barcodes]
  )

  return products.map(mapOffProduct)
}
```

### OFF nutrient mapper

```typescript
// packages/core/src/off_mapper.ts (DOM-free, RN-safe)
export interface OffNutrients {
  calories_per_100g: number | null
  protein_per_100g: number | null
  carbs_per_100g: number | null
  fat_per_100g: number | null
}

export function mapOffNutrients(nutriments: Record<string, unknown>): OffNutrients {
  return {
    // OFF uses 'energy-kcal_100g' OR 'energy_100g' (kJ); prefer kcal
    calories_per_100g: extractNumber(nutriments, 'energy-kcal_100g')
      ?? kJToKcal(extractNumber(nutriments, 'energy_100g')),
    protein_per_100g:  extractNumber(nutriments, 'proteins_100g'),
    carbs_per_100g:    extractNumber(nutriments, 'carbohydrates_100g'),
    fat_per_100g:      extractNumber(nutriments, 'fat_100g'),
  }
}

function extractNumber(obj: Record<string, unknown>, key: string): number | null {
  const v = obj[key]
  if (typeof v === 'number' && isFinite(v) && v >= 0) return v
  if (typeof v === 'string') {
    const n = parseFloat(v)
    if (isFinite(n) && n >= 0) return n
  }
  return null
}

function kJToKcal(kj: number | null): number | null {
  return kj !== null ? Math.round((kj / 4.184) * 10) / 10 : null
}

// A product is "complete" if all four macros are present and non-null
export function hasCompleteNutrients(product: { nutriments?: Record<string, unknown> }): boolean {
  if (!product.nutriments) return false
  const mapped = mapOffNutrients(product.nutriments)
  return Object.values(mapped).every(v => v !== null)
}
```

**Why `packages/core`?** The RN app may eventually display OFF products. The mapper contains
no Node.js or DOM APIs, so it belongs in core.

### Barcode lookup

```typescript
async function lookupBarcode(barcode: string): Promise<OffResult | null> {
  // 1. Check product cache
  const cached = await db.query(
    `SELECT * FROM off_foods_cache
     WHERE barcode = $1 AND cached_at > NOW() - INTERVAL '7 days'`,
    [barcode]
  )
  if (cached.rows.length > 0) return rowToOffResult(cached.rows[0])

  // 2. OFF API barcode lookup
  const url = `https://world.openfoodfacts.org/api/v2/product/${encodeURIComponent(barcode)}.json`
    + `?fields=code,product_name,brands,nutriments`
  const res = await fetch(url, { signal: AbortSignal.timeout(5000) })
  if (res.status === 404) return null
  if (!res.ok) throw new Error(`OFF barcode lookup failed: ${res.status}`)

  const data = await res.json()
  if (data.status !== 1 || !data.product) return null

  const product = data.product
  if (!hasCompleteNutrients(product)) {
    // Return partial data with null nutrients; caller decides how to handle
    return { ...mapOffProduct(product), incomplete: true }
  }

  await upsertOffProduct(product)
  return mapOffProduct(product)
}
```

### Rate limiting awareness

OFF allows anonymous requests at approximately 100 req/s. Our cache-first strategy means
typical usage makes at most 1 live request per unique search term per 7 days. No explicit
rate limiter needed in Phase 8; log a warning if we detect > 50 live OFF requests per minute
(a sign the cache is not working).

## Edge cases & failure modes

| Scenario | Handling |
|---|---|
| OFF API times out (> 5s) | `AbortSignal.timeout(5000)` throws; caught in resolver; return PERSONAL + USDA only with `off_error` |
| OFF API returns HTTP 5xx | Same catch block — graceful degradation |
| OFF API returns product with missing macros | `hasCompleteNutrients` returns false; product excluded from results |
| OFF product has energy only in kJ | `kJToKcal` converts; mapped to `calories_per_100g` |
| Barcode not in OFF | `lookupBarcode` returns null; API returns 404 |
| Search returns 0 OFF results | Deduped OFF list is empty; response still valid |
| Normalised name collision (dedup removes valid OFF entry) | Acceptable false positive — PERSONAL/USDA entry is preferred anyway |
| Cache is stale (> 7 days) | Lazy eviction deletes on read; re-fetches from OFF |
| Multiple concurrent requests for same uncached query | Both may hit OFF simultaneously; both upsert the same data. Harmless due to `ON CONFLICT DO UPDATE`. |
| OFF product name is empty string | `hasCompleteNutrients` does not check name; add a guard: skip products where `product_name` is falsy |
| Very long search term (> 200 chars) | Truncate `q` to 100 chars before building URL and cache key |
| `off_search_cache` grows unboundedly | Lazy eviction handles TTL; a periodic cleanup query can be added in Phase 9 ops work |

## Open questions

1. **OFF API stability:** OFF is a volunteer-run project. The API occasionally has downtime
   or schema changes. Our graceful degradation handles downtime; schema changes in `nutriments`
   may silently produce nulls (detected by `hasCompleteNutrients`). Should we monitor for
   increased null rates? Not in Phase 8 — add to Phase 9 ops checklist.

2. **Cache TTL of 7 days:** OFF data is crowd-edited and can change. 7 days balances freshness
   vs. rate limits. Should brand-name products (where data is stable) have a longer TTL than
   generic items? Over-engineering for Phase 8.

3. **Barcode lookup for Phase 9:** `GET /off/food/:barcode` is stubbed. When a camera-scan UI
   is added, the endpoint is ready. Confirm Phase 9 scope before assuming this is in.

4. **Deduplication quality:** Lowercase + trim catches exact duplicates but not near-duplicates
   ("Chicken Breast" vs. "Chicken Breast, Cooked"). This means some near-duplicates will appear
   in results. Good enough for Phase 8 — fuzzy matching can be added later if user feedback
   calls for it.

5. **Privacy:** OFF search queries are sent to OFF's servers as plain text. Users searching for
   personal food names (e.g., a custom recipe name) should use the PERSONAL layer, not search.
   No PII in typical search terms, but worth noting in the privacy policy.

6. **`page_size=10` in OFF search:** We fetch 10 results per search. Is this enough? More means
   more OFF API load and longer response times. 10 is a reasonable first pass; increase if user
   search results feel thin.
